# Multi-warehouse-inventory-synchronization-system

## 多仓库库存同步系统（演进式设计）

背景：某电商公司业务快速发展，初期只有一个仓库，随后扩张到多仓库，并需要支持复杂的库存策略。本题目模拟一个库存系统从0到1再到复杂分布式系统的演进过程，每个阶段引入新的业务需求和架构挑战。你需要逐步设计并实现满足当前阶段要求的系统，同时考虑未来的可扩展性。

通用业务实体定义（贯穿所有阶段）
商品（Product）：包含属性：商品ID（SKU）、商品名称、规格（如颜色、尺寸）、类别、重量、体积。

仓库（Warehouse）：包含属性：仓库ID、仓库名称、所在地（国家/城市）、类型（中心仓/前置仓）、联系人、联系方式。

库存（Inventory）：包含属性：商品ID、仓库ID、可用库存数量、物理库存数量、锁定库存数量、安全库存阈值、上次盘点时间。

库存变动记录（InventoryLog）：包含属性：日志ID、商品ID、仓库ID、变动前数量、变动后数量、变动类型（入库/出库/调拨出/调拨入/锁定/解锁）、操作人、操作时间、关联单据号（如订单号、调拨单号）。

调拨单（TransferOrder）：包含属性：调拨单号、源仓库ID、目标仓库ID、商品ID、数量、状态（待审核/执行中/已完成/取消）、调拨规则（如优先成本）、创建时间、完成时间。

### 阶段1：单体库存管理（MVP）

目标：实现单仓库基础库存管理，支持核心库存操作，快速上线验证。

业务需求
商品管理：支持添加商品基本信息（SKU、名称、规格）。

仓库管理：当前只有一个仓库，仓库信息可配置（如名称、地址）。

入库操作：采购入库时增加库存，需记录入库单号、供应商等信息（模拟）。

出库操作：订单发货时扣减库存，需验证库存是否充足，不足则拒绝。

库存查询：按商品查询当前可用库存。

库存变动日志：所有入库/出库操作记录详细日志，便于追溯。

约束与假设
单机部署，使用内存存储或简单文件存储（如JSON文件），无需持久化数据库。

单线程操作，不考虑高并发，使用简单的锁（如lock语句）保证线程安全。

无外部系统集成，所有操作通过控制台命令模拟。

用户权限简单：区分管理员（可入库/出库）和普通用户（仅查询）。

数据模型简化
商品列表：Dictionary<string, Product>

库存表：Dictionary<string, int> 键为SKU，值为库存数量（因为只有一个仓库）。

日志表：List<InventoryLog> 内存列表。

业务变动模拟（阶段内迭代）
变动1：增加“安全库存预警”需求：管理员希望当库存低于某个阈值时，控制台输出警告。需在每次出库后检查。

变动2：增加“库存锁定”机制：用户下单时先锁定库存，支付成功后才真正扣减；若超时未支付则解锁。需引入锁定库存字段和定时任务（简单模拟）。

架构关注点
三层架构：控制台交互层（解析命令）、业务服务层（库存服务）、数据访问层（内存仓储）。

使用仓储模式抽象数据访问，便于后续替换为数据库。

利用依赖注入管理服务依赖（即使简单，也要体现）。

日志模块：可独立为服务，记录到控制台和文件。

组织影响
开发团队为1-2人，快速迭代。

需与采购、仓储部门沟通，明确操作流程和数据准确性。

### 阶段2：多仓库与阈值预警

目标：支持多个仓库独立管理库存，引入预警机制和手动调拨。

新业务需求
多仓库支持：库存数据按仓库独立存储，所有操作需指定仓库。

仓库管理：增加、修改仓库信息（支持多个）。

入库/出库：指定仓库进行操作。

安全库存阈值：每个商品在每个仓库可单独设置阈值。当库存低于阈值时，触发预警事件（控制台输出并记录日志）。

手动调拨：仓储人员可创建调拨单，将某商品从源仓库调拨到目标仓库（需保证源仓库有足够库存）。调拨过程需经过“调拨出”和“调拨入”两步，中间状态为“调拨中”。

报表需求：支持按仓库、商品汇总库存，并导出（简单控制台输出）。

约束与假设
数据持久化：需使用数据库（如SQLite或本地MySQL）存储仓库、商品、库存、日志、调拨单。

多用户操作，需考虑并发冲突（如同时出库），使用数据库行锁或乐观锁（版本号）。

预警采用事件驱动：库存变动后发布“库存变动事件”，事件处理器检查阈值并触发预警。

预警方式：模拟发送邮件（控制台输出“发送邮件至仓库管理员”），后续可扩展为真实邮件。

数据模型演进
新增仓库表。

库存表增加WarehouseId作为复合主键，增加SafeThreshold字段。

调拨单表。

业务变动模拟
变动1：调拨规则变更：调拨需要审批流程，新增“待审核”状态，审批通过后才执行扣减。

变动2：新增“批次管理”：商品可能有生产日期和批次，入库时需指定批次，出库时遵循先进先出（FIFO）规则。库存表需支持批次维度。

变动3：预警方式扩展：除了邮件，还需支持短信、站内信，使用策略模式封装通知方式。

架构关注点
引入事件总线（简单内存实现），实现观察者模式解耦库存服务和预警模块。

使用仓储模式+工作单元（Unit of Work）管理事务（如调拨操作需保证两个仓库库存同时更新）。

调拨逻辑使用状态模式管理调拨单状态。

开始考虑服务分层：应用服务、领域服务、基础设施。

组织影响
团队扩大，可能需要2-3人，明确分工（库存核心、预警、报表）。

需要和仓储部门对接调拨流程，可能涉及审批系统集成。

引入数据库管理员角色（DBA）协助设计表结构和索引。

### 阶段3：自动调拨与策略引擎

目标：实现预警后自动生成调拨单，并支持灵活配置的调拨策略，提升库存周转效率。

新业务需求
自动调拨：当某仓库某商品库存低于阈值时，系统自动根据预设规则从其他仓库调拨商品补货，无需人工干预。

调拨策略可配置：支持多种规则，如：

就近原则：选择地理距离最近的仓库。

成本优先：选择调拨成本最低的仓库（考虑运输成本、关税）。

时效优先：选择预计送达时间最短的仓库。

综合评分：加权计算。

策略可动态修改：通过配置中心（模拟）调整规则参数和优先级。

调拨任务异步处理：生成调拨单后，通过消息队列将任务发送给“调拨执行器”异步处理，避免阻塞主流程。

分布式锁：防止并发情况下重复生成调拨单（同一商品在同一仓库的预警可能同时触发多次）。

调拨失败处理：若源仓库库存不足或其他原因导致调拨失败，需记录失败原因并尝试其他源仓库或发送人工警报。

约束与假设
系统需集成配置中心（如Apollo、Nacos的简单模拟，可用本地配置文件+监听文件变化实现）。

使用消息队列（如RabbitMQ或Kafka的模拟，可用内存队列+后台任务代替）解耦预警和调拨执行。

需要分布式锁（如基于Redis的RedLock模拟，本地可用ConcurrentDictionary+SemaphoreSlim模拟）。

调拨策略引擎需支持热插拔，使用策略模式+工厂模式。

业务变动模拟
变动1：新增“预售”模式：用户提前下单预售商品，需预留库存。预留库存不计入可用库存，但需保证未来能发货。引入“预留库存”概念，预售订单占用预留库存，待商品入库后自动扣减。

变动2：安全库存阈值可动态调整：业务部门希望根据销售预测自动调整阈值，需集成预测系统（模拟）。引入阈值调整事件，重新计算预警。

变动3：调拨规则增加“考虑在途库存”：调拨决策时需考虑其他调拨单在途的商品，避免重复调拨。需维护在途库存数据。

架构关注点
引入消息驱动架构：事件发布-订阅模式升级为消息队列，确保可靠性。

调拨执行器作为独立的消费者，可能需支持重试和死信队列。

策略引擎可基于规则引擎（如NRules）或解释器模式实现。

增加分布式锁组件，封装锁获取和释放逻辑。

开始考虑服务拆分：库存核心服务、预警服务、调拨决策服务、调拨执行服务可以独立部署吗？阶段内仍可单体，但模块化清晰。

组织影响
需要与供应链、物流部门合作定义调拨成本、时效模型。

团队可能需要专门的业务分析师（BA）梳理规则。

引入运维人员管理消息队列、配置中心。

### 阶段4：分布式库存与最终一致性

目标：业务量爆发，库存服务需拆分为微服务，支持高并发扣减，并保证分布式事务最终一致性。

新业务需求
高并发扣减：秒杀场景下，库存扣减QPS达数万，需使用缓存（Redis）提升性能。

库存服务拆分：

商品服务：管理商品基本信息。

仓库服务：管理仓库信息。

库存服务：管理库存数量，提供扣减/查询接口。

库存预留服务：管理预售、订单锁定库存。

分布式事务：下单扣库存涉及订单服务、库存服务、支付服务，需保证最终一致性。采用Saga模式或TCC（Try-Confirm-Cancel）。

幂等性：所有接口支持幂等，防止重复请求。

分库分表：库存数据按商品ID分片，缓解单库压力。

缓存一致性：Redis库存与数据库库存保持最终一致，采用Cache-Aside模式，并监听数据库变更刷新缓存。

约束与假设
使用分布式缓存Redis（可模拟本地缓存+过期时间）。

微服务架构：每个服务独立数据库，通过RESTful API或gRPC通信。可简化用Process模拟多个服务。

分布式事务框架：可使用本地消息表+消息队列实现最终一致性（如RocketMQ事务消息）。

服务注册与发现：简单模拟（如静态配置）。

需要分布式ID生成器（如雪花算法）。

业务变动模拟
变动1：新增“多级库存”概念：仓库分为中心仓和前置仓，中心仓调拨到前置仓，前置仓直发客户。调拨流程变为两级。

变动2：引入“滞销品清理”：对于长期未动销的商品，自动生成清仓建议，推送给运营。

变动3：库存可视化：需要实时大屏展示全国库存热力图，要求数据延迟不超过1分钟。需引入实时流计算（如模拟Kafka + Flink），但可简化为定时推送。

架构关注点
微服务拆分策略：按领域划分，每个服务独立演进。

分布式事务方案选择：根据业务场景（扣库存要求强一致性还是最终一致）决定。

缓存设计：如何保证缓存与数据库一致？如何避免缓存穿透、雪崩？

服务间通信：同步（HTTP/gRPC）与异步（消息）的合理使用。

容错设计：断路器、重试、超时、降级（使用Polly库模拟）。

数据一致性：分布式事务中的补偿机制设计。

组织影响
团队按服务拆分，形成多个跨职能小团队（如库存团队、订单团队）。

需要定义服务间的SLA和契约测试。

引入DevOps文化，自动化部署、监控。

架构师需协调各团队，确保整体架构一致性。

### 阶段5：全球化与合规

目标：业务扩展到海外，需支持多区域部署，遵守各地数据法规，并实现全球库存协同。

新业务需求
多区域部署：每个区域（如北美、欧洲、亚太）独立部署一套库存服务，数据隔离。

数据合规：欧洲数据必须留在欧洲，不能跨境传输。需设计数据分区策略。

全球库存视图：运营中心需要看到全球库存总量，但具体细节按区域权限访问。通过事件中心汇总各区域库存变化，生成全局视图（数据脱敏）。

本地化策略：不同国家税率、关税影响调拨成本，策略引擎需支持区域特定的成本模型。

跨国调拨：支持从一国仓库调拨到另一国仓库，需处理海关、关税、国际物流。调拨流程需引入关务模块。

GDPR合规：支持用户删除个人数据（如操作人信息），需设计数据匿名化或清理机制。

约束与假设
每个区域独立数据库，区域间通过消息队列或数据同步工具（如Canal）同步必要数据（仅限脱敏后的汇总数据）。

引入数据脱敏模块，对敏感信息（如操作人IP、手机号）进行加密或掩码。

使用API网关进行区域路由和权限校验。

关务模块可模拟，需集成外部报关系统（设计防腐层）。

业务变动模拟
变动1：某国突发政策变更，要求所有库存数据实时上报给政府监管系统。需快速适配新接口。

变动2：并购了一家海外物流公司，需将其仓库系统对接进来，使用防腐层隔离对方老旧系统的差异。

变动3：全球促销活动：同一个商品在全球各区域同时降价，需保证库存预留一致性（但数据隔离，需通过总部分配额度）。

架构关注点
多活/主备架构设计：每个区域独立可用，但需处理跨区域数据一致性（通常不强求）。

防腐层（Anti-Corruption Layer）：隔离外部系统（如海关、物流）的变化。

数据合规：设计数据分类分级，实现数据最小化收集。

全球化部署的监控和故障定位：分布式追踪（如Jaeger模拟）。

配置管理：区域特定的配置（如税率、货币）集中管理。

组织影响
团队进一步分化为区域团队和平台团队，平台团队提供共享组件（如配置中心、事件中心）。

需要法律顾问参与定义合规要求。

跨国沟通协作成为挑战，文档和API需多语言支持。

总结：这个演进式题目展示了库存系统从单体到全球分布式系统的完整路径，每个阶段都引入了真实的业务复杂性和架构挑战。你需要针对每个阶段设计合理的解决方案，并考虑未来的扩展性。在实现过程中，可以重点关注设计模式的应用、模块化拆分、以及如何应对不断变化的业务需求。
